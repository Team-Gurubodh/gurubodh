# Semantic Chunking Requirements

## Purpose

This note explains how Gurubodh's semantic chunking capability should work.
After testing and manual confirmation, these chunks and metadata associated with each chunk should become the source of truth for chunks.
Presently, `gurubodh` implementation already divides a "subject" into multiple "chapters". Metadata associated with each chapter should remain separate from that for the chunks. The "chapter text" and "chunk text" together with their individual metadata gives complete flexibility in terms of how `apps/gurubodh-web` and `apps/gurubodh-chat` applications will be implemented in the near future.


## Current Capability Already Present

- The existing `semantic_chunking` module already wraps `BAAI/bge-m3` in order to generate paragraphs from raw text supplied from the chapter text.
- The current module lives under:

```text
tools/gurubodh-cli/gurubodh/ml/semantic_chunking/
```

- `BAAI/bge-m3` is an embedding model. Its job is to convert text into vectors.
- It already performs the main steps needed for embedding-assisted semantic
paragraphing:

1. Split the raw chapter text into sentence-like units.
2. Build contextual sentence windows.
3. Embed those windows with `BAAI/bge-m3` through `sentence-transformers`.
4. Compare neighboring window embeddings using cosine distance.
5. Treat unusually large semantic distance as a likely topic shift.
6. Use `threshold_percentile` to choose candidate breakpoints.
7. Convert breakpoints into chunks.
8. Merge undersized chunks using `min_chars`.
9. Return chunks with text, sentence counts, character counts, and sentence
   ranges.

Tested parameters like `threshold_percentile` and `min_chars` produces 
reasonable Hindi paragraphs. It does so by answering questions like:

- Are these two neighboring sentence windows semantically close?
- Did the topic shift sharply between one sentence region and the next?
- Are these sentences likely part of the same idea?

## Why Spans, as detailed in Task-014 Do Not Matter

Task-014 originally proposed asking Sarvam for paragraph spans instead of chunks. 
The reason was Sarvam does not have embedding APIs / endpoints yet. 

When `BAAI/bge-m3` based semantic chunker returns paragraphs, downstream rendering can concatenate paragraphs with some simple markup or markdown to display an entire chapter in a legible way.

So, Task-014 should remove all requirements related to spans or getting Sarvam to do any kind of transformation for the purpose of generating paragraphs or chunks from chapters. Instead it should be noted that Sarvam will be used later in the RAG layer of `apps/gurubodh-chat` application.

## Requirements For Task-014 Rewrite

### 1. Rename The Direction

Task-014 should no longer be framed as Sarvam-backed paragraph segmentation since 
the intended provider is local semantic chunking.

Recommended naming:

```text
Embedding-Assisted Semantic Paragraph Segmentation
```

### 2. Replace The Existing Span Contract

The durable metadata should record important data about chunks in JSON schema. For example:

- enabled: true / false
- provider: "semantic-chunking"
- model: "BAAI/bge-m3"
- threshold_percentile
- min_chars
- window_size
- batch_size
- normalize_embeddings
- optional device selection such as `cpu`, `mps`, or `cuda`
- chunking strategy/version
- "status": "segmented" or something similar
- sentence_count
- char_count
- "source_text_sha256": "..."
- "chunk_text_sha256": "..." 
  
Important objective is to "Keep Tunable Parameters Explicit", because these settings affect output. If they change, paragraph spans may change.

It should be noted that the above properties is just an example. These should be replaced if better metadata properties are decided later.

### 3. Add Provider Boundary Before Wider Integration

Direct model loading should be isolated.

Allowed:

```text
Semantic chunking provider loads BAAI/bge-m3.
```

Avoid:

```text
Chapter preparation code directly instantiates SentenceTransformer("BAAI/bge-m3").
```

The `prep-subject` pipeline within `gurubodh-cli` should depend on semantic chunking interface for paragraphing, not on a particular model library.

Recommended conceptual interface:

```python
class ParagraphSegmenter:
    def segment(self, text: str, source_name: str | None = None) -> ParagraphSegmentation:
        ...
```

Then the first implementation can be:

```text
SemanticChunkingParagraphSegmenter
```

### 4. Avoid Implicit Model Downloads

The implementation should not download or load `BAAI/bge-m3` at import time.

Model loading should happen only when paragraph segmentation is invoked and
enabled.

This matters because many CLI commands should remain lightweight and should not
require ML dependencies or model files unless they actually use semantic
chunking.

### 5. Standardize Model Cache Configuration

Define the convention now. Recommended environment variables:

```text
GURUBODH_MODEL_CACHE_DIR
HF_HOME
SENTENCE_TRANSFORMERS_HOME
```

Recommended behavior:

- if `GURUBODH_MODEL_CACHE_DIR` is set, configure local model cache paths from
  it;
- otherwise use normal Hugging Face defaults;
- document that Docker will later mount the same cache path as a shared volume.

This saves disk space later and prevents each component from inventing its own
model-cache behavior.


### 6. Validate Before Storing

Before writing chunk metadata, validate:

- chunks are ordered;
- chunks do not overlap;
- chunks are within source bounds;
- chunks are non-empty;
- Any other reasonable validation that should be added


### 7. Adjust Task-014 Carry-Forward Items

When rewriting Task-014, keep these concepts:

- canonical raw `.txt` remains authoritative;
- paragraph /chunk metadata stores validated spans;
- checksum ties spans to source text;
- disabled-by-default config;
- `continue_on_error`;
- status values such as `disabled`, `segmented`, `skipped-unchanged`, `failed`;
- audit reports;
- progress reporting;

Drop these Sarvam-specific items:

- Sarvam prompt;
- Sarvam structured response schema;
- Sarvam HTTP caller;
- `SARVAM_API_KEY`;
- Sarvam retry/throttle policy;
- Sarvam token usage;
- output-token economy justification.

## Other pointers for modifying Task-014

A revised Task-014 should consider following points:

- Paragraph segmentation / chunking can be enabled per conversion job.
- Omitted config keeps segmentation disabled.
- Semantic chunking is invoked only when segmentation is enabled.
- The implementation uses configured model/provider values, not scattered
  hard-coded calls.
- `BAAI/bge-m3` is loaded only behind a provider/segmenter boundary.
- The configured model cache convention is documented.
- Successful chapter metadata stores spans, checksum, provider, model,
  parameters, status, and diagnostics.
- Failed segmentation / chunking records a failure without breaking raw chapter output
  when `continue_on_error` is true.
- Existing DOCX conversion and chapter text output remain unchanged.
- Tests cover  chunking, metadata writing, disabled config, failure behavior, and checksum reuse etc.
