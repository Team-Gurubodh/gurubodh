# 0009 — Embedding Model and LLM Provider (Phase 3)

## Status

Proposed

## Context

The RAG layer needs an embedding model to vectorize content and queries, and
an LLM to generate grounded answers from retrieved content. The source text is in devanagari, with repeated usage of important terms and concepts from the spiritual domain, more specifically, Sanatana Dharma and/or Indian Knowledge System. Present LLM models are mostly trained in English language without much training in the domain of Sanatana Dharma and Indian Knowledge System.

The content-preparation workspace also has local semantic chunking experiments
that use `BAAI/bge-m3` through `sentence-transformers`. This model may be useful
for local dense embeddings and future RAG experiments, but the architecture
should not make a specific embedding model the integration boundary. Embedding
models can change over time, and vectors generated by different models,
dimensions, modes, or normalization settings are not safely comparable.

## Decision

**Recommended (pending confirmation):** Use Amazon Bedrock for both — Titan
Embeddings for vectorization and a Claude model for answer generation — to
keep the RAG stack AWS-native and consistent with the platform's overall
hosting decision (ADR-0003).

For local content-preparation and RAG prototyping, `BAAI/bge-m3` may be used as
the default local embedding model. This is a model choice, not an architecture
boundary.

All embedding generation must go through a provider boundary instead of calling
a specific model directly from ingestion, retrieval, or CMS-facing code. The
provider boundary must expose the model identity, embedding mode, vector
dimension, normalization behavior, and batch encoding behavior needed by the
pipeline.

The first pgvector-based RAG phase should use dense vectors only. BGE-M3 sparse
and multi-vector modes may be added later behind the same provider boundary or
as additional provider modes once dense retrieval is working end to end.

Stored embeddings and vector index records must include enough metadata to
identify which vectors are mutually comparable:

- embedding provider;
- model name;
- embedding mode, such as `dense`, `sparse`, or `colbert`;
- embedding dimension when applicable;
- normalization setting;
- embedding/index schema version;
- chunking strategy and version;
- source content and chunk identity.

Chunks and source content identity must remain separate from embedding identity.
Changing the embedding model, mode, dimension, or normalization setting requires
regenerating derived embeddings and rebuilding the affected vector index, but it
should not require changing canonical content artifacts or stable chunk
identifiers.

## Consequences

**Positive**
- A single AWS bill and IAM model governs access to both the embedding model
  and the LLM, rather than managing a separate vendor relationship and API key.
- Data stays within the AWS account boundary already used for the rest of the
  platform.
- Local BGE-M3 experiments can proceed without making downstream RAG code depend
  directly on BGE-M3-specific APIs.
- Dense-first retrieval keeps the first pgvector RAG implementation simple while
  preserving a path to sparse or multi-vector retrieval later.
- Model swaps become predictable rebuild operations because embedding/index
  metadata records the provider, model, mode, dimension, and version.

**Negative**
- Tied to whichever models Bedrock offers and supports, which may lag behind
  using a provider's API directly.
- Switching embedding models later requires re-embedding all existing content,
  so this choice has real switching costs once content volume grows.
- The LLM may not understand what the user is actually seeking when a question is asked and thus may not give correct answers.
- The provider boundary adds a small amount of abstraction before there are
  multiple production embedding providers.
- Supporting sparse or multi-vector modes later will require additional storage,
  indexing, retrieval, and evaluation design beyond the dense pgvector MVP.

**Alternatives Considered**
- **OpenAI or Cohere APIs directly** — potentially best-in-class models for
  embeddings or generation, but introduces a non-AWS vendor with separate
  billing, security review, and data-handling considerations.
- **Hard-code BGE-M3 directly into RAG pipelines** — fast for local experiments,
  but would make future model swaps expensive and spread model-specific
  assumptions across ingestion, retrieval, and storage code.
